# 밴픽 + 섬멸전 전용 런처 구조 기획서 (수정안)

> 이 문서는 밴픽 기반 섬멸전 모드를 **전용 외부 런처 + BPM 전용 섬멸전 맵 내부 밴픽** 구조로 구현하기 위한 설계 문서이다.  
> 외부 런처, 통합 확장 모드, BPM 전용 섬멸전 맵, 유닛/맵 데이터베이스, 옵저버/심판 지원을 포함한다.

> 수정 기준: 기존 `BanpickLauncher.SC2Map` + `Next Map` + `Bank 전달` 구조를 제거하고,  
> 외부 런처는 설치/업데이트/실행만 담당하며 실제 밴픽은 BPM 전용 섬멸전 맵 내부에서 처리하는 구조로 정리했다.

---

## 1. 개요

### 1.1 프로젝트 목표

- 밴픽 시스템을 포함한 1v1 섬멸전 모드를  
  **전용 외부 런처 + BPM 전용 섬멸전 맵 내부 밴픽** 구조로 구현한다.
- 외부 런처는 Mass Recall식 설치/업데이트/실행 도구로 사용하되,  
  실제 밴픽 판정과 밴 적용은 SC2 맵 내부 트리거에서 처리한다.
- 기존 섬멸전의 `로딩 후 3, 2, 1 즉시 시작` 흐름은 직접 막지 않고,  
  게임 시작 직후 시작 유닛/자원/입력을 잠근 뒤 밴픽을 진행하는 방식으로 우회한다.
- 공허의 유산 기본 유닛 풀과 캠페인+스타1 확장 유닛 풀을 모두 지원하되,  
  로비 Game Attribute 또는 인게임 호스트/심판 UI에서 모드를 선택한다.
- 기존 편의성 확장 모드(`ConvenienceBase`)의 기능을  
  **현재 프로젝트의 통합 확장 모드 내부로 흡수**하여,  
  모드 간 의존성/충돌 가능성을 제거한다.
- 편의 기능 및 밴픽/밸런스에 사용되는 **유닛 데이터(아이콘 경로, 버튼, 업그레이드, 메타 정보)**를  
  에디터 상의 “데이터베이스(메타 데이터 테이블)” 형태로 정의하고,  
  트리거/코드는 이 DB를 참조해 동작하도록 만든다.
- 여러 확장 모드(`ConvenienceBase`, `BanPickSys`, `CampaignBalance`)를  
  **하나의 최종 확장 모드로 통합**하되,  
  내부적으로는 기능별 폴더/트리거 라이브러리/데이터 카테고리로  
  명확히 분리해 유지보수성을 확보한다.
- 1v1 플레이 + 옵저버(관전자) 0~N명, 선택적으로 심판 슬롯을 지원한다.

### 1.2 전체 사용자 플로우

1. **외부 런처 실행**
   - 사용자가 프로젝트 전용 런처(EXE)를 실행한다.
   - 런처는 SC2 설치 경로, `Mods/`, `Maps/` 파일, 버전을 확인한다.
   - 필요 시 GitHub Release 등에서 최신 패키지를 다운로드/설치한다.

2. **맵 실행 또는 사용자 지정 게임 입장**
   - 로컬/테스트 실행:
     - 런처가 `SC2Switcher_x64.exe` 또는 `SC2Switcher.exe`에  
       `BPM_1v1_*.SC2Map` 경로를 인자로 넘겨 직접 실행한다.
   - 멀티플레이 실행:
     - 런처는 설치/업데이트 보장까지만 담당한다.
     - 실제 게임은 Battle.net 사용자 지정 게임/아케이드에서  
       BPM 전용 섬멸전 맵을 선택해 시작한다.

3. **SC2 기본 로딩 및 3, 2, 1**
   - SC2는 일반 맵처럼 로딩 후 기본 카운트다운을 진행한다.
   - 이 카운트다운은 엔진 차원에서 막지 않는다.

4. **게임 시작 직후 밴픽 잠금 단계**
   - 맵 초기화 직후:
     - 시작 유닛을 생성하지 않거나 즉시 숨김/정지한다.
     - 자원, 생산, 명령, 카메라 입력을 잠근다.
     - 전체 화면 밴픽/옵션 UI를 표시한다.

5. **모드/옵션 확정 및 밴픽 진행**
   - 옵션:
     - 유닛 풀 버전(UnitPoolMode: 공허 바닐라 / 캠페인+스타1),
     - 밴 개수(BanCount: 0 / 1 / 3 / 5),
     - 선공/후공, 공개/쉐도우 밴 등.
   - 롤(LoL) 스타일 풀스크린 밴픽 UI에서 밴을 진행한다.

6. **밴 결과 적용 및 가짜 로딩 연출**
   - 밴 결과는 같은 게임 세션 내 전역 변수/구조체에 저장한다.
   - `FinalDisabledUnits`를 계산하고 Ban 업그레이드/Requirement를 적용한다.
   - 화면에 “Loading Match…” 또는 자체 카운트다운을 표시한다.

7. **섬멸전 시작**
   - 각 플레이어 시작 위치에 본진/일꾼을 생성한다.
   - 초기 자원, 시야, 카메라, 동맹/관전자 상태를 설정한다.
   - 입력 잠금을 해제하고 실제 섬멸전 경기를 시작한다.

### 1.3 대상 환경

- 게임 모드:
  - 1v1 섬멸전(플레이어 2명),
  - 옵저버(관전자) 0~N명,
  - 선택적으로 심판 슬롯 1명.
- 실행 환경:
  - Battle.net 사용자 지정 게임/Arcade,
  - 로컬 에디터 `Test Document`,
  - 외부 런처를 통한 로컬 `.SC2Map` 직접 실행.
- 맵:
  - 공허의 유산 래더 맵을 기반으로 한 BPM 전용 맵 파일,
  - 프로젝트 전용 섬멸전 맵.
- 핵심 제약:
  - BPM 전용 섬멸전 맵에서 `Next Map`으로 실제 섬멸전 맵을 전환하는 구조는 사용하지 않는다.
  - 밴픽 결과를 다른 맵으로 넘기기 위한 Bank 의존 구조도 핵심 경로에서 제외한다.

## 2. 시스템 구조

### 2.1 최종 구조 결정

본 프로젝트의 시스템 구조는 기존 초안의  
`외부 런처 → SC2 런처 맵 → 밴픽 → Bank 저장 → Next Map → 섬멸전 맵` 흐름을 사용하지 않는다.

최종 구조는 다음과 같다.

```text
외부 런처
→ 설치/업데이트/파일 검증/맵 실행
→ BPM 전용 섬멸전 맵 로드
→ 게임 시작 직후 플레이 잠금
→ 인게임 밴픽
→ 밴 결과 적용
→ 시작 유닛 생성
→ 섬멸전 시작
```

핵심 원칙:

- 외부 런처는 **배포·실행·버전 통일 도구**이다.
- 밴픽 판정, 밴 결과 동기화, 생산 제한 적용은 **SC2 맵 내부 트리거**가 담당한다.
- 밴 결과를 다른 맵으로 전달하기 위한 `Bank`/`Next Map` 구조는 핵심 경로에서 제거한다.
- `3, 2, 1`을 막는 대신, 그 직후 실제 게임 진행을 잠그고 밴픽 UI를 띄운다.

### 2.2 구성 요소 및 역할

#### 2.2.1 외부 런처 클라이언트 – `BPM Launcher.exe`

- 역할
  - SC2 설치 경로 자동 검출 및 설정 저장.
  - 프로젝트에 필요한 `Mods/` 및 `Maps/` 폴더 구조, 파일 존재 여부 검사.
  - GitHub Release 등에서 최신 패키지 다운로드/업데이트.
  - 설치 파일 manifest 기록 및 안전한 제거.
  - 로컬 테스트용으로 `SC2Switcher_x64.exe` 또는 `SC2Switcher.exe`를 호출해  
    특정 `BPM_1v1_*.SC2Map`을 직접 실행.
- 하지 않는 역할
  - 밴픽 결과 확정.
  - 멀티플레이 판정/동기화.
  - 각 플레이어의 금지 유닛을 외부에서 강제 전달.
  - Battle.net 대기실을 완전히 대체하는 매치메이킹.

#### 2.2.2 통합 확장 모드 – `BPM_Core.SC2Mod` 또는 `BPM_Core.SC2Mod`

- 기존 모드 통합:
  - `ConvenienceBase`의 편의 기능,
  - `BanPickSys`의 밴픽 UI/로직,
  - `CampaignBalance`의 유닛 풀/캠페인/스타1/밸런스 조정.
- 기능
  - 밴픽 시스템:
    - 밴 라운드/타이머/역할 그룹 스왑,
    - 밴 후보 풀 생성,
    - 인게임 HUD(ME/ENEMY).
  - 밸런스/유닛 풀:
    - 공허 바닐라 유닛 풀,
    - 캠페인+스타1 확장 유닛 풀,
    - Ban 업그레이드/Requirement,
    - 특수 밸런스 조정.
  - 편의 기능:
    - 자원, 카메라, 핫키, 옵저버 UI, 인터페이스 편의 기능.
  - 데이터베이스:
    - `UnitMeta`, `UpgradeMeta`, `ButtonMeta`, `MapMeta` 등 메타 테이블.

#### 2.2.3 BPM 전용 섬멸전 맵 – `BPM_1v1_*.SC2Map`

- 역할
  - 실제 섬멸전 경기가 진행되는 맵.
  - 공허의 유산 래더 맵/커스텀 맵을 기반으로 하되,  
    `BPM_Core.SC2Mod`를 의존성으로 가진다.
  - 게임 시작 직후 밴픽 잠금 상태로 진입한다.
  - 밴픽 완료 후 시작 유닛/자원/카메라를 생성하고 실제 경기를 시작한다.
- 각 맵에 포함되는 것
  - 지형,
  - 시작 위치,
  - 자원 배치,
  - 젤나가 타워/감시탑 등 맵 오브젝트,
  - 맵별 초기화용 최소 트리거.
- 각 맵에 넣지 않는 것
  - 공통 밴픽 UI/로직,
  - 확장 유닛 데이터,
  - 공통 Requirement/업그레이드 토큰,
  - 공통 편의 기능.

#### 2.2.4 선택적 로컬 테스트 맵

- `BPM_1v1_TestMap.SC2Map`
- 런처가 기본으로 실행할 수 있는 개발/검증용 맵.
- 실제 래더 맵 추가 전:
  - 밴픽 UI,
  - Ban 업그레이드 적용,
  - 시작 잠금/해제,
  - 옵저버/심판 동작
  을 검증하는 용도로 사용한다.

### 2.3 실행 모드별 동작

#### 2.3.1 로컬 테스트

```text
BPM Launcher.exe
→ 파일/버전 확인
→ SC2Switcher_x64.exe "Maps\BanPickMode\BPM_1v1_TestMap.SC2Map"
→ SC2 로딩
→ 게임 시작 직후 밴픽
→ 섬멸전 시작
```

- Codex/자동화/에디터 테스트에 적합하다.
- 런처에서 실행할 맵을 직접 선택할 수 있다.
- 싱글 테스트 또는 AI/테스트 플레이어 기준 검증에 유리하다.

#### 2.3.2 Battle.net 사용자 지정 게임

```text
BPM Launcher.exe
→ 설치/업데이트 보장
→ 사용자가 Battle.net에서 BPM 맵 생성/입장
→ 로비 Game Attribute 또는 인게임 Host UI에서 룰 선택
→ 게임 시작
→ 게임 시작 직후 밴픽
→ 섬멸전 시작
```

- 실제 멀티플레이 기준 권장 구조다.
- 런처는 버전 통일과 파일 설치를 보장한다.
- 밴픽 결과는 같은 SC2 게임 세션 내부 전역 변수로만 관리한다.

### 2.4 패키지 구조

초기 MVP 패키지:

```text
BPM_Release.zip
├─ Mods
│  └─ BPM_Core.SC2Mod
└─ Maps
   └─ BanPickMode
      └─ BPM_1v1_TestMap.SC2Map
```

확장 후 패키지:

```text
BPM_Release.zip
├─ Mods
│  ├─ BPM_Core.SC2Mod
│  ├─ BPM_Assets.SC2Mod        # 선택
│  └─ BPM_ExtendedUnits.SC2Mod # 선택
└─ Maps
   └─ BanPickMode
      ├─ BPM_1v1_TestMap.SC2Map
      ├─ BPM_1v1_Gresvan.SC2Map
      ├─ BPM_1v1_AncientCistern.SC2Map
      └─ BPM_1v1_Custom_*.SC2Map
```

권장 원칙:

- 처음에는 `BPM_Core.SC2Mod` 하나로 시작한다.
- 유닛/아이콘/음성/모델 파일이 커지면 `BPM_Assets.SC2Mod`로 분리한다.
- 모든 섬멸전 맵은 같은 `BPM_Core.SC2Mod`를 참조한다.

### 2.5 맵 풀 구조

기존 초안의 `MapPool → Next Map 로드` 방식은 제거한다.

수정된 MapPool의 역할:

- 런처 UI에서 로컬 실행 가능한 맵 목록을 보여준다.
- 인게임 UI에서 “현재 맵 정보”를 표시한다.
- Battle.net 사용자 지정 게임에서는 실제 선택된 맵이 이미 로드되어 있으므로,  
  MapPool은 동적 맵 전환용이 아니라 **메타 정보 표시 및 버전 검증용**이다.

필드 예시:

- `MapId`
- `FileName`
- `DisplayNameKey`
- `MiniMapImage`
- `MapSize`
- `DescriptionKey`
- `IsOfficialLadder`
- `RequiredCoreVersion`

### 2.6 유닛/아이콘/버튼/업그레이드 데이터베이스화

이 부분은 기존 기획을 유지한다.

- `UnitMeta`
  - UnitId, DisplayNameKey, IconPath, Race, TechTier, Role, IsWorker, IsBanEligible 등.
- `UpgradeMeta`
  - UpgradeId, RelatedUnitId, BanLockUpgradeId, IconPath 등.
- `ButtonMeta`
  - ButtonId, UnitId, AbilityId, BanRequirementId 등.

사용처:

- 밴픽 UI 유닛 그리드 생성.
- 공허 바닐라/캠페인+스타1 후보 풀 필터링.
- Ban 업그레이드/Requirement 적용.
- HUD 아이콘/툴팁 표시.

### 2.7 플레이어/옵저버/심판 구조

기본 구조:

- Player 1: PlayerA
- Player 2: PlayerB
- Player 3: Referee 또는 Observer
- Player 4~N: Observer

권장 처리:

- 심판 슬롯을 사용할 경우:
  - Player 3 = Referee.
  - 맵/옵션 확정 및 밴픽 시작 권한을 가진다.
- 심판 슬롯을 사용하지 않을 경우:
  - HostPlayer = Player 1 기본.
  - Player 1 또는 방장이 맵 내 옵션/밴픽 시작을 담당한다.
- 옵저버:
  - 밴픽 UI 관전 가능.
  - 클릭/확정 입력은 무시.
  - 인게임에서는 전체 시야 및 옵저버 HUD를 사용한다.

### 2.8 시작 잠금 구조

SC2 기본 로딩 후 3, 2, 1 카운트다운은 그대로 진행된다.  
대신 맵은 다음 상태로 시작한다.

```text
Map Initialization
→ 시작 유닛 미생성 또는 즉시 정지/숨김
→ 카메라 고정
→ 명령/생산/자원 조작 비활성
→ 밴픽 UI 표시
```

밴픽 완료 후:

```text
밴 결과 확정
→ FinalDisabledUnits 계산
→ Ban 업그레이드 지급
→ Requirement 반영
→ 가짜 로딩/자체 카운트다운
→ 시작 유닛 생성
→ 초기 자원 지급
→ 카메라 이동
→ 입력 잠금 해제
```

이 방식이 현재 프로젝트의 핵심 시스템 구조다.

## 3. 사용자 플로우 상세

> 이 장에서는 “외부 런처 → BPM 전용 섬멸전 맵 → 인게임 밴픽 → 섬멸전 시작”까지의  
> 전체 흐름을 단계별로 정리한다. 플레이어, 옵저버, 심판 시점을 모두 포함한다.

---

### 3.1 전체 단계 개요

1. 외부 런처 실행
2. 파일/버전 검사 및 필요 시 업데이트
3. BPM 전용 섬멸전 맵 실행 또는 Battle.net 사용자 지정 게임 입장
4. SC2 기본 로딩 및 3, 2, 1 진행
5. 게임 시작 직후 플레이 잠금
6. 모드/옵션 확인
7. 밴픽 진행
8. 밴 결과 적용
9. 가짜 로딩/자체 카운트다운
10. 시작 유닛/자원 생성
11. 경기 진행 + 관전

---

### 3.2 외부 런처 단계

**목표:** 사용자가 실행 환경을 준비하고, 로컬 테스트 시 바로 BPM 맵을 실행할 수 있게 한다.

- 사용자 행동
  - 외부 런처(EXE)를 실행한다.
  - “설치/업데이트” 또는 “Play” 버튼을 클릭한다.
- 런처 동작
  - SC2 설치 경로 확인.
  - `BPM_Core.SC2Mod`, `BPM_1v1_*.SC2Map` 존재 여부 확인.
  - 필요 시 GitHub Release에서 최신 ZIP 다운로드.
  - 설치된 파일 manifest와 버전 정보를 저장한다.
- 실행 분기
  - 로컬 테스트:
    - 런처에서 선택한 `.SC2Map`을 `SC2Switcher_x64.exe`로 직접 실행.
  - 멀티플레이:
    - 런처는 설치/업데이트 완료 상태만 보장하고,
    - 사용자는 Battle.net 사용자 지정 게임에서 BPM 맵을 생성/입장한다.

---

### 3.3 SC2 로딩 및 초기 잠금 단계

**목표:** 기본 카운트다운 후에도 실제 게임이 바로 진행되지 않게 한다.

- SC2 동작
  - 일반 섬멸전 맵처럼 로딩 후 3, 2, 1 카운트다운이 진행된다.
- 맵 초기화
  - 시작 유닛은 미리 배치하지 않거나, 생성 직후 제어 불가능 상태로 둔다.
  - 플레이어 카메라를 밴픽 UI용 위치 또는 검은 화면/전용 배경으로 고정한다.
  - 자원/명령/생산/유닛 이동 등 플레이 입력을 제한한다.
  - 전체 화면 Dialog 기반 밴픽/옵션 UI를 표시한다.

---

### 3.4 모드/옵션 확인 단계

**목표:** 이번 경기에 사용할 밴픽 룰을 확정한다.

- 설정 항목
  - UnitPoolMode: 공허 기본 / 캠페인+스타1
  - BanCount: 0 / 1 / 3 / 5
  - FirstMover: Player 1 / Player 2 / Random
  - ShadowBan 사용 여부
  - 옵저버 쉐도우 밴 공개 방식
- 권한
  - 심판이 있으면 심판이 확정.
  - 심판이 없으면 HostPlayer가 확정.
  - PlayerA/B는 Ready 또는 동의 버튼을 사용할 수 있다.
  - 옵저버는 읽기 전용.
- 동기화
  - 설정은 현재 게임 세션의 전역 변수에 저장한다.
  - Bank를 핵심 전달 수단으로 사용하지 않는다.

---

### 3.5 밴픽 진행 단계

**목표:** 설정된 룰에 따라 두 플레이어가 밴을 수행한다.

- 화면 구성
  - 상단: 라운드, 타이머, 현재 차례, 밴 타입.
  - 좌측/우측: Player 1/2 밴 리스트.
  - 중앙: 유닛 아이콘 그리드.
  - 하단: 선택 미리보기, 확인 버튼, 안내 문구.
- 라운드 흐름
  - `BanRounds[]`에 정의된 순서대로 진행한다.
  - 각 라운드는 `Owner`, `Target`, `Visibility`를 가진다.
  - 해당 라운드의 Owner만 유닛 선택/확정 가능.
  - 타이머가 끝나면 미선택 패스 또는 옵션에 따라 랜덤 밴.
- 후보 필터링
  - `UnitMeta.IsBanEligible == true`
  - 현재 UnitPoolMode에 포함된 유닛
  - 일꾼/핵심 시작 유닛 제외
  - 이미 밴된 유닛 제외
  - 역할 그룹 스왑/그룹 밴 규칙 반영

---

### 3.6 밴 결과 적용 단계

**목표:** 선택된 밴이 실제 생산/변태/업그레이드/패널 호출에 반영되게 한다.

- 내부 구조
  - `PublicBansAgainst[player]`
  - `ShadowBansAgainst[player]`
  - `SelfBans[player]`
  - `FinalDisabledUnits[player]`
- 적용 방식
  - `FinalDisabledUnits[player]` 계산.
  - 각 유닛에 대응하는 `BPM_Ban_UnitX` 업그레이드를 해당 플레이어에게 지급.
  - 생산 버튼/변태 버튼/업그레이드/패널/용병 호출은  
    Requirement에서 해당 Ban 업그레이드가 없을 때만 사용 가능하도록 설정한다.

---

### 3.7 가짜 로딩 및 섬멸전 시작 단계

```text
밴픽 완료
→ Loading Match... 표시
→ Ban 업그레이드 적용
→ 시작 위치/종족/자원 확인
→ 본진/일꾼 생성
→ 카메라 이동
→ 자체 3, 2, 1 또는 Ready 표시
→ 입력 잠금 해제
```

중요:

- 이 단계는 실제 맵 로딩이 아니다.
- 같은 `.SC2Map` 안에서 밴픽 UI를 닫고 섬멸전 상태로 전환하는 것이다.

---

### 3.8 경기 진행 + 관전

- 플레이어
  - 밴된 유닛/능력은 생산·사용 불가.
  - 상단 HUD에서 ME/ENEMY 밴 정보를 확인.
  - 승리 조건은 일반 섬멸전과 동일.
- 옵저버
  - 전체 시야 및 옵저버 UI 사용.
  - 밴 HUD는 토너먼트 모드에서는 양측 실제 밴을 모두 표시.
- 심판
  - 인게임 판정 UI는 후순위.
  - 기본적으로 밴픽 단계의 시작 권한과 운영 확인 역할을 담당한다.

## 4. 외부 런처 설계

> 이 장에서는 Mass Recall식 **외부 런처(EXE)**의 역할을 정의한다.  
> 런처는 게임 룰을 판정하는 주체가 아니라, 설치/업데이트/실행을 담당하는 클라이언트다.

---

### 4.1 런처의 역할 범위

#### 4.1.1 필수 역할

- SC2 설치 경로 자동 탐색 및 수동 지정.
- `Mods/`와 `Maps/` 파일 설치 여부 확인.
- GitHub Release 기반 패키지 다운로드/업데이트.
- SHA256 또는 버전 JSON 기반 파일 검증.
- 설치 manifest 저장 및 안전한 제거.
- 로컬 테스트용 `.SC2Map` 직접 실행.
- 패치노트/버전 정보 표시.

#### 4.1.2 하지 않는 역할

- 밴픽 결과를 외부에서 결정하지 않는다.
- 각 플레이어의 금지 유닛을 Bank로 강제 주입하지 않는다.
- 멀티플레이 동기화의 권위 주체가 되지 않는다.
- Battle.net 대기실을 완전히 대체하지 않는다.

---

### 4.2 SC2 실행 방식

권장 실행 방식:

```text
SC2Root\Support64\SC2Switcher_x64.exe "SC2Root\Maps\BanPickMode\BPM_1v1_TestMap.SC2Map"
```

대체 경로:

```text
SC2Root\Support\SC2Switcher.exe
SC2Root\StarCraft II.exe
```

우선순위:

1. `Support64\SC2Switcher_x64.exe`
2. `Support\SC2Switcher.exe`
3. `StarCraft II.exe`

주의:

- 로컬 직접 실행은 테스트/싱글/AI 검증에 강점이 있다.
- 실제 멀티플레이는 Battle.net 사용자 지정 게임에서 BPM 맵을 실행하는 구조를 기본으로 한다.

---

### 4.3 파일/경로 관리

기본 설치 구조:

```text
StarCraft II
├─ Mods
│  └─ BPM_Core.SC2Mod
└─ Maps
   └─ BanPickMode
      ├─ BPM_1v1_TestMap.SC2Map
      ├─ BPM_1v1_Gresvan.SC2Map
      └─ BPM_1v1_*.SC2Map
```

런처 로컬 상태:

```text
%LOCALAPPDATA%\SC2BanPickModeLauncher
├─ state.txt
├─ install-path.txt
└─ manifest.txt
```

---

### 4.4 업데이트 패키지

GitHub Release 예시:

```text
BPM_Release.zip
BPM_latest.json
```

ZIP 설치 규칙:

- ZIP 안의 `Mods/` 폴더는 SC2Root `Mods/`에 복사.
- ZIP 안의 `Maps/` 폴더는 SC2Root `Maps/`에 복사.
- ZIP Slip 방지를 위해 압축 해제 경로가 캐시 폴더 밖으로 나가지 않는지 검사.
- 제거 시 manifest에 기록된 파일만 삭제한다.
- 삭제 대상은 반드시 SC2Root `Mods/` 또는 `Maps/` 하위로 제한한다.

---

### 4.5 런처 UI

MVP UI:

- 상태 라벨:
  - 설치 안 됨 / 설치됨 / 업데이트 가능 / 오류
- 버튼:
  - 설치
  - 업데이트
  - 파일 확인
  - 제거
  - 실행
- 맵 선택:
  - `BPM_1v1_TestMap`
  - 설치된 `BPM_1v1_*.SC2Map` 목록
- 옵션:
  - SC2 경로 수동 선택
  - 캐시 삭제
  - 사용자 설정 초기화

---

### 4.6 런처 ↔ SC2 데이터 교환 범위

기본 방침:

- 런처는 SC2에 **매치 핵심 데이터**를 넘기지 않는다.
- 런처가 넘길 수 있는 것은 다음 정도로 제한한다.
  - 로컬 테스트용 실행 맵 경로.
  - 최근 실행 맵.
  - UI 언어/화면 옵션 같은 사용자 편의 설정.
  - 외부 매치 ID 같은 비권위 메타 정보.

밴픽/밴 결과/승패와 직결되는 정보는 SC2 맵 내부에서 확정한다.

## 5. BPM 전용 섬멸전 맵 내 밴픽 단계 설계

> 기존 초안의 `BPM_1v1_TestMap.SC2Map`은 제거한다.  
> 밴픽 UI와 대기/옵션/심판 흐름은 실제 섬멸전 맵 내부의 시작 전 단계로 통합한다.

---

### 5.1 맵 개요

- 맵 이름 예시:
  - `BPM_1v1_TestMap.SC2Map`
  - `BPM_1v1_Gresvan.SC2Map`
  - `BPM_1v1_AncientCistern.SC2Map`
- 의존성:
  - `BPM_Core.SC2Mod`
- 기본 동작:
  1. SC2 로딩 및 기본 3, 2, 1 진행.
  2. 게임 시작 직후 모든 플레이어를 시작 잠금 상태로 둔다.
  3. 밴픽/옵션 UI를 표시한다.
  4. 밴픽 완료 후 Ban 적용.
  5. 시작 유닛/자원 생성.
  6. 실제 섬멸전 시작.

---

### 5.2 초기화: 시작 잠금 및 UI 생성

- 기본 섬멸전 시작 유닛을 맵에 미리 배치하지 않는다.
- 필요한 경우 기본 배치 유닛을 즉시 제거/숨김 처리한다.
- 플레이어 카메라를 전용 배경 또는 시작 위치 밖으로 고정한다.
- 명령/생산/선택 입력을 제한한다.
- `RootDialog`를 생성하고 옵션/밴픽 UI를 표시한다.

단계 상태 변수:

- `BPMStage`
  - 0 = 초기 잠금
  - 1 = 옵션 확인
  - 2 = 밴픽 진행
  - 3 = 밴 결과 적용
  - 4 = 매치 시작 연출
  - 5 = 섬멸전 진행 중

---

### 5.3 옵션 확인 UI

- 유닛 풀 버전:
  - 공허의 유산 기본
  - 캠페인+스타1 유닛 포함
- 밴 개수:
  - 0 / 1 / 3 / 5
- 쉐도우 밴:
  - 사용 / 미사용
- 선공:
  - Player 1 / Player 2 / Random
- 권한:
  - 심판이 있으면 심판만 확정.
  - 심판이 없으면 HostPlayer가 확정.
  - PlayerA/B는 Ready 또는 동의 버튼 사용 가능.

---

### 5.4 밴픽 UI/로직

- 상단:
  - 타이머, 라운드, 현재 차례, 밴 타입.
- 좌측:
  - Player 1 밴 리스트.
- 우측:
  - Player 2 밴 리스트.
- 중앙:
  - 유닛 아이콘 그리드.
- 하단:
  - 선택 미리보기, 확인 버튼, 안내 문구.

권한:

- 현재 라운드 Owner만 선택 가능.
- 상대/옵저버/심판은 관전 또는 관리 UI만 사용.
- 이미 밴된 유닛, 밴 불가 유닛, 현재 풀에 없는 유닛은 비활성화한다.

---

### 5.5 밴 결과 적용

```text
PublicBansAgainst
+ ShadowBansAgainst
+ SelfBans
→ FinalDisabledUnits
→ BPM_Ban_UnitX 업그레이드 지급
→ Requirement 반영
```

적용 대상:

- 생산 버튼
- 변태/진화 버튼
- 연구 업그레이드
- 캠페인 패널
- 용병/특수 소환
- 기타 우회 생산 경로

---

### 5.6 매치 시작 연출

- 밴픽 UI를 닫는다.
- 전체 화면에 “Loading Match…” 또는 자체 카운트다운을 표시한다.
- 시작 위치에 본진/일꾼을 생성한다.
- 초기 자원/시야/카메라를 설정한다.
- 플레이어 입력 잠금을 해제한다.

주의:

- 이 단계는 실제 맵 로드가 아니다.
- `Game - Set Next Map` 또는 `Game - End Game`을 사용하지 않는다.

## 6. 섬멸전 맵 설계 (`BPM_1v1_*.SC2Map`)

> 이 장에서는 실제 전장 역할을 하는 BPM 전용 섬멸전 맵의 요구사항과  
> 초기화/밴 적용/HUD/옵저버 처리 방식을 정의한다.

---

### 6.1 맵 목록 및 역할

- 파일명 예시:
  - `BPM_1v1_TestMap.SC2Map`
  - `BPM_1v1_Gresvan.SC2Map`
  - `BPM_1v1_AncientCistern.SC2Map`
- 역할:
  - 각 맵은 1v1 섬멸전 경기를 위한 실제 전장이다.
  - 밴픽/밸런스/편의 기능은 `BPM_Core.SC2Mod`에서 통일 관리한다.
  - 맵 자체는 지형/시작 위치/자원/맵별 오브젝트만 가진다.

---

### 6.2 공통 요구 사항

- Player 1, 2:
  - 실제 참가자.
- Player 3:
  - 설정에 따라 심판 또는 옵저버.
- Player 4~N:
  - 옵저버.
- 지형:
  - 표준 1v1 섬멸전 레이아웃.
- 자원:
  - 본진/앞마당/추가 멀티 자원 배치.
- 승리 조건:
  - 기본 섬멸전 승리 조건을 사용한다.

---

### 6.3 초기화 로직

`Map Initialization`에서 다음을 수행한다.

1. 플레이어 역할 판정.
2. 기본 시작 유닛 생성 방지 또는 제거.
3. 카메라/입력/자원/명령 잠금.
4. 로비 Game Attribute 또는 기본값으로 옵션 초기화.
5. 옵션 확인 UI 표시.
6. 밴픽 시작.
7. 밴 결과 적용.
8. 시작 유닛 생성.
9. 입력 잠금 해제.

---

### 6.4 밴 결과 적용

- `FinalDisabledUnits[player]`를 계산한다.
- 각 유닛에 대응하는 `BPM_Ban_UnitX` 업그레이드를 해당 플레이어에게 지급한다.
- Data Requirement는 다음 조건을 가진다.

```text
사용 가능 조건 =
  필요한 테크 조건 충족
  AND CountUpgrade(BPM_Ban_UnitX, Player) == 0
```

적용 경로:

- 일반 생산
- 변태/진화
- 연구
- 캠페인 패널
- 용병
- 특수 소환
- 자동 생성/트리거 생성 우회 경로

---

### 6.5 시작 유닛 생성

밴픽 완료 후 종족에 따라 시작 유닛을 생성한다.

예시:

- Terran:
  - Command Center
  - SCV 12기
- Zerg:
  - Hatchery
  - Drone 12기
  - Overlord
- Protoss:
  - Nexus
  - Probe 12기

주의:

- 종족 Random 처리 시 밴픽 후보 풀과 실제 종족 확정 시점이 충돌하지 않도록 한다.
- 권장 방식:
  - 실제 종족은 게임 시작 전 확정.
  - Random이면 밴픽 UI 진입 전에 내부적으로 종족을 확정한다.

---

### 6.6 인게임 HUD

- 상단 중앙에 밴 HUD 생성.
- Player 기준:
  - ME = 자신에게 적용된 밴.
  - ENEMY = 상대에게 적용된 밴.
- 옵저버 기준:
  - 기본은 양측 밴 모두 표시.
  - 토너먼트 모드에서는 쉐도우 밴 실제 유닛도 표시 가능.

## 7. 데이터 교환 설계

> 기존 초안의 핵심 데이터 교환 구조였던  
> `BPM 전용 섬멸전 맵 Bank 저장 → 섬멸전 맵 Bank 로드` 방식은 제거한다.  
> 수정된 구조에서는 밴픽과 섬멸전이 같은 `.SC2Map` 세션 안에서 이어진다.

---

### 7.1 기본 방침

- 매치 핵심 데이터는 Bank가 아니라 **현재 게임 세션의 전역 변수/구조체**로 관리한다.
- Bank는 다음 용도로만 사용한다.
  - 사용자 UI 설정.
  - 최근 선택 옵션.
  - 디버그/개발 로그.
  - 외부 매치 ID 같은 비권위 메타 정보.
- 밴 결과, 금지 유닛, 선공, 승패 판정 등은 Bank를 권위 소스로 사용하지 않는다.

---

### 7.2 인게임 전역 구조체

필수 구조체/변수:

- `CfgUnitPoolMode`
- `CfgBanCount`
- `CfgFirstMover`
- `PlayerA`
- `PlayerB`
- `RefereePlayer`
- `ObserverPlayers[]`
- `PublicBansAgainst[player][]`
- `ShadowBansAgainst[player][]`
- `SelfBans[player][]`
- `FinalDisabledUnits[player][]`
- `BPMStage`
- `BanRounds[]`
- `CurrentBanRoundIndex`

이 데이터는 밴픽 완료 후 그대로 밴 적용 단계에서 사용된다.

---

### 7.3 로비 Game Attribute

Battle.net 사용자 지정 게임에서 미리 선택 가능한 값:

- `BPM_ContentMode`
  - `LOTV_ONLY`
  - `EXTENDED_UNITS`
- `BPM_BanCount`
  - `0`
  - `1`
  - `3`
  - `5`
- `BPM_ShadowBan`
  - `ON`
  - `OFF`
- `BPM_ObserverReveal`
  - `PLAYER_VIEW`
  - `TOURNAMENT_REVEAL`

맵 초기화 시 이 값을 읽고 기본 설정으로 사용한다.

---

### 7.4 외부 런처 설정 파일

런처가 저장할 수 있는 비권위 설정:

```json
{
  "last_map": "BPM_1v1_TestMap.SC2Map",
  "preferred_content_mode": "EXTENDED_UNITS",
  "preferred_ban_count": 3,
  "ui_language": "ko-KR"
}
```

주의:

- 이 값은 로컬 편의 기본값일 뿐이다.
- 멀티플레이에서 최종 룰은 SC2 로비/맵 내부 Host/심판 UI에서 확정한다.

---

### 7.5 Bank 사용 범위

선택적 Bank:

- 파일명:
  - `BPMUserSettings.SC2Bank`
- 섹션:
  - `[User]`
  - `[UI]`
  - `[Debug]`

저장 예시:

- 마지막 HUD 모드.
- 옵저버 HUD 표시 여부.
- 디버그 HUD On/Off.
- 마지막 사용 언어.

사용하지 않는 것:

- `PublicBans`
- `ShadowBans`
- `FinalDisabledUnits`
- 매치 승패
- 밴픽 확정 결과

## 8. 모드/데이터 정의 (통합 확장 모드 `BPM_Core.SC2Mod`)

> 이 장에서는 통합 확장 모드의 **기능 구조(트리거)**와  
> **데이터 구조(유닛/업그레이드/UI/DB)**를 정의한다.  
> 기존 `ConvenienceBase`, `BanPickSys`, `CampaignBalance`를 하나로 합친 형태를 기준으로 한다.

---

### 8.1 모드 개요

- 모드 이름(가칭): `BPM_Core.SC2Mod`
- 목적:
  - 편의 기능(QoL),
  - 밴픽 시스템,
  - 유닛 풀/캠페인+스타1/밸런스 조정,
  - 메타 데이터(DB) 관리
  를 **단일 모드**에서 제공한다.
- 의존성:
  - 기본 라이브러리: `Liberty`, `Swarm`, `Void`
  - 멀티/캠페인: `Liberty Campaign`, `Swarm Campaign`, `Void Campaign`, `Void Multiplayer` (필요 범위에 맞게)
- 사용 대상:
  - `BPM_1v1_TestMap.SC2Map` (BPM 전용 섬멸전 맵),
  - `BPM_1v1_*.SC2Map` (섬멸전 인게임 맵).

---

### 8.2 트리거 구조

#### 8.2.1 폴더 구조 개요

- `Triggers/Convenience/` – 편의 기능
- `Triggers/BanPick/` – 밴픽 로직/UI/데이터
- `Triggers/Balance/` – 유닛 풀/캠페인·스타1/밸런스
- `Triggers/Lobby/` – BPM 전용 섬멸전 맵 대기실/맵 선택/옵저버
- `Triggers/Init/` – 공통 초기화/유틸리티

#### 8.2.2 Triggers/Convenience

기존 `ConvenienceBase` 기능에서 필요한 것만 선별해 이 안에 포팅한다.

- 예시 서브 라이브러리:
  - `Conv_Resources` : 시작 자원/작업자 수, 수동/자동 자원 관련 QoL.
  - `Conv_Camera` : 카메라 초기 위치, 관전 시점 보정.
  - `Conv_UI` : 핫키, 정보 표시, 간단 카운터/타이머 등.
  - `Conv_Misc` : 기타 게임 품질개선 기능.

원칙:

- **유닛/업그레이드 ID를 직접 하드코딩하지 않고**,  
  필요 시 `UnitMeta`/`UpgradeMeta` DB를 참조하도록 단계적으로 개선한다.
- 섬멸전/캠페인 공통으로 사용 가능한 기능만 남기고,  
  프로젝트와 관련 없는 기능은 제거 or 비활성화.

#### 8.2.3 Triggers/BanPick

밴픽 관련 로직과 UI를 분리/모듈화한다.

- 서브 라이브러리 예시:
  - `BP_Core`  
    - 공통 타입/상수/전역 변수 정의.
    - `CfgUnitPoolMode`, `CfgBanCount`, `FinalDisabledUnits[]` 등.
  - `BP_Rounds`  
    - Ban 라운드 시퀀스 생성/진행.
    - Owner/Target/Visibility 정의.
  - `BP_UnitPool`  
    - `UnitPoolMode`에 따른 유닛 후보 리스트 생성.
    - 일꾼/밴 불가 유닛 필터링.
  - `BP_LauncherUI`  
    - BPM 전용 섬멸전 맵용 밴픽 UI (풀스크린 레이아웃 제어).
  - `BP_IngameApply`  
    - 섬멸전 맵에서 Ban 결과를 Ban 업그레이드/Requirement로 적용.
  - `BP_HUD`  
    - 인게임 HUD(ME/ENEMY 토글), 아이콘 리스트 갱신.

#### 8.2.4 Triggers/Balance

유닛 풀/캠페인+스타1/밸런스 조정 관련 로직.

- 서브 라이브러리 예시:
  - `BP_UnitPoolMeta`
    - Version A/B에서 어떤 유닛이 포함/제외되는지 메타 정보 관리.
  - `BP_CampaignUnits`
    - 캠페인/스타1 유닛 활성화/비활성 트리거(필요 시).
  - `BP_RoleGroups`
    - 역할 그룹 정의(예: Zerg Spellcasters, Protoss Spellcasters).
    - Version B에서 그룹 스왑/GroupBanFlag 반영 로직.
  - `BP_BalanceTweaks`
    - 리버 등 특정 유닛의 능력/사거리/쿨다운 조정(트리거 필요 시).

#### 8.2.5 Triggers/Lobby

BPM 전용 섬멸전 맵에서 사용하는 대기실/맵 선택/옵저버용 트리거.

- 서브 라이브러리 예시:
  - `Lobby_Init`
    - BPM 전용 섬멸전 맵 초기화, 기본 UI 숨김, 루트 다이얼로그 생성.
  - `Lobby_PlayerSlots`
    - 플레이어/옵저버 슬롯 표기/Ready 로직.
  - `Lobby_MapSelect`
    - 맵 풀 표시/선택, CurrentMapIndex 관리.
  - `Lobby_Options`
    - UnitPoolMode, BanCount, 기타 룰 옵션 설정.
  - `Lobby_StageFlow`
    - Stage(대기실→옵션→밴픽→완료) 전환 관리.
  - `Lobby_StateCommit`
    - 현재 게임 세션 전역 상태 확정/검증.

#### 8.2.6 Triggers/Init & Util

- `Init_Global`
  - 모드 전역 변수 초기화, DB 로딩, 기본 값 세팅.
- `Util_UserSettings`
  - 선택적 사용자 설정 Bank 읽기/쓰기 공통 함수.
- `Util_Strings`
  - 리스트 문자열 파싱/직렬화(`ParseUnitList`, `SerializeUnitList` 등).
- `Util_Player`
  - PlayerA/PlayerB ↔ Player 번호 매핑.
- `Util_Color`
  - 플레이어 색상 정보 조회 및 UI에 적용.

---

### 8.3 데이터 모듈 구조

#### 8.3.1 Units (Data/Units)

- 기본 원칙:
  - 기존 멀티/캠페인 유닛을 직접 수정하기보다는,  
    **필요한 경우에 한해 복제/부분 수정**으로 처리.
- 카테고리 예시:
  - `BP_Units_Core`
    - 프로젝트 전용 유닛(예: 테스트 유닛, 특수 토템 등).
  - `BP_Units_Campaign`
    - 캠페인 유닛 추가/수정(히페리온, 히어로 유닛 등).
  - `BP_Units_SC1`
    - 스타1 유닛(리버, 드라군, 벌쳐 등)의 SC2 버전.
  - `BP_Units_Balance`
    - 밸런스 조정된 유닛 데이터(리버 사거리/쿨타임 변경 등).

- 캠페인/스타1 유닛은:
  - `UnitMeta` DB와 연결될 수 있도록  
    일관된 UnitId 네이밍을 사용:
    - 예: `ReaverBW`, `DragoonBW`, `VultureBW`.

#### 8.3.2 Upgrades (Data/Upgrades)

- `BP_Upgrades_BanLocks`
  - 각 유닛에 대응하는 Ban 업그레이드:
    - 예: `BPM_Ban_Marine`, `BPM_Ban_ReaverBW`.
  - Player별로 업그레이드 레벨을 1로 올려 Ban 적용.
- `BP_Upgrades_Balance`
  - 밸런스용 업그레이드(특정 유닛 능력 강화/너프).
- `BP_Upgrades_Campaign`
  - 캠페인/스타1 전용 업그레이드(연구, 패널 등).

#### 8.3.3 UI (Data/UI)

- Layout/Frame 정의:
  - BPM 전용 섬멸전 맵용 UI 프레임 (대기실/맵 선택/밴픽 레이아웃).
  - 인게임 HUD 프레임(ME/ENEMY 표시).
- 아이콘:
  - 밴 HUD용 잠금/쉐도우 아이콘.
  - 특수 옵션/룰 아이콘.
- 버튼/Requirement 연결:
  - 커맨드 카드/패널/용병 버튼에 Ban Requirement 연결 시  
    필요한 UI 요소 정의.

---

### 8.4 DB 영역 (Data/DB)

**핵심 요구:**  
“유닛/업그레이드/버튼/맵” 정보가 하드코딩 대신  
**메타 데이터 레코드**로 관리되도록 한다.

#### 8.4.1 UnitMeta

- 구현 방식:
  - Record 또는 Dummy Unit/Behavior/Upgrade에  
    커스텀 필드를 사용해 메타 정보 저장.
- 필드 예시:
  - `UnitId` : SC2 유닛 ID (Record key 자체로 활용 가능)
  - `DisplayNameKey`
  - `IconPath`
  - `Race` (T/Z/P)
  - `TechTier` (1/2/3)
  - `Role` (DPS/Tank/Siege/Spell 등)
  - `IsWorker`
  - `IsBanEligible`
  - `DefaultPool_A` (Version A에서 사용 여부)
  - `DefaultPool_B` (Version B에서 사용 여부)

이 DB는:

- 런처 밴픽 UI의 유닛 그리드 구성,
- 유닛 풀 필터링,
- HUD/툴팁 표시의 근거가 된다.

#### 8.4.2 UpgradeMeta

- 필드 예시:
  - `UpgradeId`
  - `RelatedUnitId`
  - `BanLockUpgradeId` (Ban 업그레이드와 1:1, 필요 시)
  - `IconPath`
- 사용처:
  - 특정 유닛의 생산/업그레이드 잠금 제어를 일관되게 할 수 있도록 연결.

#### 8.4.3 ButtonMeta

- 필드 예시:
  - `ButtonId`
  - `UnitId` / `AbilityId`
  - `BanRequirementId`
- 사용처:
  - Ban 업그레이드와 Requirement를 연결할 때,  
    “어떤 버튼이 어떤 유닛/능력을 나타내는지”를 매핑하는 데이터.

#### 8.4.4 MapPool

- 필드 예시는 2.2에서 정의한 내용과 동일:
  - `MapId`, `FileName`, `DisplayNameKey`, `MiniMapImage`, `MapSize`, `DescriptionKey`, `IsOfficialLadder` 등.
- BPM 전용 섬멸전 맵의 맵 선택 UI 및  
  섬멸전 맵 검증(이 맵이 어떤 ID로 선택된 것인지 확인)에 사용.

---

### 8.5 기존 ConvenienceBase 코드 흡수 전략

#### 8.5.1 단계적 포팅

1. ConvenienceBase 트리거/데이터에서  
   프로젝트와 관련된 기능만 선별.
2. 기능별로 `Triggers/Convenience` 아래에 복사:
   - `Conv_Resources`, `Conv_Camera`, `Conv_UI` 등.
3. 외부 `ConvenienceBase` 모드 의존성을  
   모든 맵에서 제거하고,
   - `BPM_Core.SC2Mod`만 사용하도록 변경.

#### 8.5.2 충돌 방지

- 동일 이름/ID를 가진 유닛/업그레이드/버튼이  
  다른 모드나 캠페인과 충돌하지 않게:
  - ID에 `BP_` prefix를 붙이는 등 네이밍 규칙을 정한다.
- 기존 편의 모드에서 쓰던 ID가  
  밴픽/밸런스와 겹치지 않도록 정리.

---

### 8.6 밴픽/밸런스와 DB의 연결

#### 8.6.1 밴픽 → DB 사용

- UnitGrid 구성 시:
  - `UnitMeta`에서 `IsBanEligible = true`인 유닛만 후보로 사용.
  - `DefaultPool_A/B`와 `UnitPoolMode`에 따라 필터링.
- 라운드 대상(본인/상대)에 따라:
  - Target Race/UnitSet을 결정할 때 `UnitMeta.Race` 등을 사용.

#### 8.6.2 밸런스 → DB 사용

- Version A/B에서 유닛 풀을 구성할 때:
  - `UnitMeta.DefaultPool_A/B` 값을 기반으로 유닛 목록을 생성.
- 캠페인/스타1 유닛 토글:
  - `UnitMeta`에 `IsCampaignOnly`/`IsSC1` 플래그를 두고,
    룰/옵션에 따라 활성/비활성 제어.

#### 8.6.3 편의 기능 → DB 사용 (장기 목표)

- 편의 기능이 특정 유닛/업그레이드를 참조할 때도:
  - 직접 ID를 쓰는 대신 `UnitMeta`/`UpgradeMeta`를 조회.
- 예:
  - 특정 유닛의 생산 상황/자원 표시 기능을  
    DB 기반으로 확장하기 용이해진다.

---

### 8.7 버전 관리 / 모드 업데이트

- `BPM_Core.SC2Mod` 버전 문자열:
  - 예: `BPM_Core v0.1.0`.
- 주요 변경 시:
  - 설정 구조체 `ConfigVersion`,
  - DB Record 버전,
  - 맵/런처의 요구 버전 등을 변경 기록.
- 외부 런처에서:
  - 모드 버전 확인 후,
  - 구버전/신버전 간 호환성 안내 or 자동 업데이트 지원.

## 9. 옵저버 지원

> 이 장에서는 **옵저버(관전자)**가 보는 UI/정보 구성을 정의한다.  
> 기본적으로 **GameHeart 스타일 관전자 UI**를 기반으로 하되,  
> 양측의 밴픽 정보를 추가로 보여주는 형태로 확장한다.

---

### 9.1 목표 및 전제

- 옵저버는 **GameHeart 기반 관전자 UI**를 사용한다.
  - 생산/인구/자원/APM/인게임 타이밍 등 기존 정보 유지.
- 여기에 **양측 밴픽 결과(공개 + 쉐도우 상태)**를  
  추가 HUD로 표시한다.
- BPM 전용 섬멸전 맵의 밴픽 단계와 섬멸전 인게임 모두에서:
  - 플레이어와 거의 동일한 정보를 보되,
  - 입력(클릭/확정)은 불가.
- 플레이어 수:
  - Player 1, Player 2: 실제 플레이어.
  - Player 3~N: 옵저버(관전자).

---

### 9.2 BPM 전용 섬멸전 맵에서의 옵저버 지원

#### 9.2.1 대기실 화면

- 옵저버 리스트:
  - RightPanel에 Player 3~N의 이름/색상 표시.
- 권한:
  - Ready/맵 변경/옵션 변경/밴픽 조작 불가.
  - UI 요소는 “비활성화된 버튼” 또는 커서 변경으로 표현.
- 정보:
  - 맵 선택 및 옵션 변경 상황은 실시간으로 옵저버 화면에도 동기화.

#### 9.2.2 밴픽 화면

- 옵저버는 플레이어와 동일한 밴픽 풀스크린 UI를 본다:
  - 좌측: Player 1 밴 리스트.
  - 우측: Player 2 밴 리스트.
  - 중앙: 유닛 그리드, 상단 타이머/라운드 표시.
- 입력:
  - 유닛 그리드 클릭/확인 버튼은 비활성화.
  - 마우스 오버 툴팁(유닛 이름/설명)은 허용.
- 공개/쉐도우 표시:
  - 플레이어 화면 기준과 동일 규칙:
    - 공개 밴: 실제 유닛 아이콘 + 이름.
    - 쉐도우 밴: 잠금/마스크 아이콘, 이름 미표시.

---

### 9.3 인게임 관전자 UI – GameHeart 기반

> 이 섹션은 GameHeart 스타일의 관전자 UI를 **전제로 한 확장**이다.  
> 실제 구현에서는 GameHeart 레이아웃/트리거를 참고하거나,  
> 유사한 구조를 모방해 제작한다.

#### 9.3.1 기본 정보 표시

- 상단:
  - 양 플레이어 이름, 종족, 색상, 점수/시리즈 스코어(선택).
- 좌측/우측:
  - 인구, 자원, APM, 업그레이드, 생산 큐 등 기본 GH HUD.
- 하단:
  - 타이머, 시청자/옵저버 리스트 등.

밴픽 확장 기능은 **기본 GameHeart HUD 위에 추가 레이어**로 구현한다.

---

### 9.4 인게임 밴픽 HUD – 옵저버 확장

#### 9.4.1 HUD 위치 및 형태

- 위치:
  - 화면 상단 중단, GameHeart 팀 이름/스코어 아래/사이에 배치.
- 구조:
  - 좌측 밴 바(Team 1, Player 1)
  - 우측 밴 바(Team 2, Player 2)
- 각 밴 바:
  - 최대 N개의 슬롯(예: 5개).
  - 슬롯별:
    - 공개 밴: 유닛 아이콘.
    - 쉐도우 밴: 잠금/마스크 아이콘.
  - 밴 타입(상대/자기/쉐도우)은 아이콘 테두리/오버레이 색으로 구분.

예시:

- 왼쪽 밴 바: Player 1 관점
  - 파란 테두리: Player 1이 밴한 상대 유닛.
  - 회색 잠금 아이콘: Player 1에게 숨겨진 쉐도우 밴(실제 유닛 미공개).
- 오른쪽 밴 바: Player 2 관점(빨간 테두리).

#### 9.4.2 옵저버용 정보 범위

옵저버에게 밴 정보를 어디까지 보여줄지 두 가지 모드가 있다:

- **토너먼트 모드(완전 공개)** – 기본안
  - 옵저버는 양쪽의 **실제 밴 유닛**을 모두 볼 수 있다.
  - 즉:
    - 플레이어 화면에서는 쉐도우 밴이 잠금 아이콘으로만 보이지만,
    - 옵저버(방송)는 실제 유닛 아이콘/이름을 본다.
- **플레이어 시점 모드(부분 공개)**
  - 옵저버도 특정 플레이어 시점 그대로 본다.
  - 이 경우 쉐도우 밴은 아이콘으로 숨겨진 상태 그대로.

기획 기본값은 **토너먼트/관전 편의상 “완전 공개 모드”**로 한다.  
(설정으로 두 모드 중 선택 가능하게 확장 여지 유지)

#### 9.4.3 HUD 동작 로직

- 게임 시작 시:
  - 각 플레이어의 `FinalDisabledUnits`,  
    `PublicBans`, `ShadowBans`, `SelfBans`를 읽어 HUD 데이터 생성.
- HUD 갱신:
  - Ban은 게임 시작 전에 확정이므로,  
    인게임 중에는 HUD 내용이 변하지 않는다.
- 툴팁:
  - 아이콘 위에 마우스를 올리면:
    - 유닛 이름 + 짧은 역할 설명 표시.

---

### 9.5 옵저버 HUD 토글 및 시점 전환

#### 9.5.1 기본 시점

- 기본적으로 옵저버 HUD는 “양쪽 모두 보여주는 모드”를 사용한다:
  - 상단 좌우 밴 바에 양 팀의 밴 정보를 동시에 표시.
- GameHeart 시점 토글(예: 탭 키로 P1/P2/양쪽 전환)을 사용할 경우,
  - 밴 바 테두리 강조색을 그 시점 팀에 맞게 조정.

#### 9.5.2 세부 토글 (선택 사항)

추가로 다음과 같은 토글을 제공할 수 있다:

- `Ctrl+B` : 밴 HUD 전체 On/Off.
- `Ctrl+1` / `Ctrl+2` : Player 1/2 관점으로 밴 HUD 축소 표시  
  (예: 좌측/우측만 확대).

이는 추후 실제 관전/방송 환경 테스트 후 필요 시 구현하는 옵션으로 둔다.

---

### 9.6 옵저버용 데이터 동기화

#### 9.6.1 인게임 전역 상태 기반 동기화

- 옵저버는 Bank가 아니라 현재 게임 세션의 밴 구조체를 참조한다.
- 밴픽 완료 시:
  - 플레이어용 Ban 구조체를 생성한다.
  - 옵저버 HUD용 데이터 구조도 함께 생성한다.
    - `ObserverBans_Player1`
    - `ObserverBans_Player2`
- “완전 공개 모드”에서는:
  - 옵저버용 구조체에 **쉐도우 밴의 실제 유닛 ID**를 넣는다.
- “플레이어 시점 모드”에서는:
  - 플레이어 구조체를 그대로 참조해 보여준다.

#### 9.6.2 게임 내 지연/재접속

- 옵저버가 중간 합류/재접속해도:
  - `Map Initialization`이 끝난 후이므로,
  - HUD 생성 트리거는 `Player - Player joins game` 이벤트에서도  
    한 번 더 실행해 최신 상태를 그려준다.

---

### 9.7 예외/에러 처리 (옵저버 관련)

- 밴 구조체가 비어 있거나 초기화되지 않은 경우:
  - HUD를 숨기고 기본 GameHeart UI만 사용.
- PlayerA/B ID가 잘못된 경우:
  - Ban HUD를 비활성화하고, 로그에만 기록.
- 옵저버가 많을 때:
  - HUD는 동일하게 유지,  
    옵저버 수가 많아도 표시 정보는 변하지 않음.

---

### 9.8 요약

- 런처/밴픽 단계:
  - 옵저버는 **플레이어와 동일한 화면**에서 밴픽 과정을 관전하되,
    조작은 불가.
- 인게임:
  - GameHeart 스타일 관전자 UI를 기본으로 사용하면서,
  - 화면 상단에 **양측 밴픽 결과 HUD**를 추가해  
    관전자/시청자가 밴 전략을 한눈에 이해할 수 있게 한다.
- 쉐도우 밴 공개 범위:
  - 기본값은 옵저버에게만 완전 공개하는 “토너먼트 모드”,
  - 필요하면 플레이어 시점과 동일하게 숨기는 모드도 옵션으로 지원.

## 10. 멀티/동기화 고려 사항

> 이 장에서는 1v1 + 옵저버 + 심판 슬롯을 포함한  
> 멀티플레이 동작 규칙과 동기화/판정 로직을 정의한다.  
> 수정된 구조에서는 밴픽과 섬멸전이 같은 맵 세션 안에서 이어지므로,  
> Bank/Next Map 기반 전달 동기화는 사용하지 않는다.

---

### 10.1 플레이어 역할/슬롯 구조

- Player 1: 참가자 A (`Role = PlayerA`)
- Player 2: 참가자 B (`Role = PlayerB`)
- Player 3: 심판 또는 옵저버 (`Role = Referee` or `Observer`)
- Player 4~N: 옵저버 (`Role = Observer`)

역할에 따른 권한:

- PlayerA/B:
  - Ready/동의 버튼 가능.
  - 자기 밴픽 차례에만 유닛 선택 가능.
- Referee:
  - 맵 내부 옵션 확정 권한.
  - 밴픽 시작/재설정 권한.
  - 필요 시 진행 중단/재시작 요청 버튼 사용.
- Observer:
  - 옵션/밴픽/경기 관전만 가능.
  - 확정 입력은 모두 무시한다.

---

### 10.2 Ready 및 시작 조건

- `ReadyA == true`
- `ReadyB == true`
- 옵션 값이 유효함
- Host 또는 Referee가 시작을 확정함

심판이 없는 경우:

- `HostPlayer`가 시작 권한을 가진다.
- 기본 Host는 Player 1.
- Player 1이 나가면 Player 2로 권한을 넘긴다.

심판이 있는 경우:

- Player 1,2는 Ready만 가능.
- 심판만 옵션 확정 및 밴픽 시작 가능.

---

### 10.3 옵션/밴픽 동기화

- 권위 데이터는 현재 게임 세션의 전역 변수에 둔다.
- 모든 옵션 변경은 Host/Referee 입력 이벤트에서만 발생한다.
- UI는 전역 변수 값을 기준으로 재렌더링한다.
- 각 밴 라운드의 Owner/Target/Visibility는 모든 클라이언트에서 같은 `BanRounds[]`를 사용한다.
- 타이머는 트리거 기준으로 감소시키고, UI만 각 플레이어에게 갱신한다.

---

### 10.4 밴픽 결과 동기화

- 밴 확정 시점에 다음 구조체를 즉시 갱신한다.
  - `PublicBansAgainst`
  - `ShadowBansAgainst`
  - `SelfBans`
  - `FinalDisabledUnits`
- 같은 맵 세션 안에서 바로 Ban 업그레이드를 지급하므로  
  다른 맵으로 결과를 넘기는 동기화 문제가 없다.
- Bank는 밴 결과의 권위 소스로 쓰지 않는다.

---

### 10.5 연결 끊김 처리

#### 10.5.1 밴픽/시작 전 단계

- PlayerA/B가 나가면:
  - 밴픽을 중단한다.
  - Ready 상태를 해제한다.
  - 남은 플레이어/심판에게 중단 메시지를 표시한다.
- 기본 안전 규칙:
  - 공식 경기 모드에서는 튕긴 플레이어 패배 처리 가능.
  - 친선/테스트 모드에서는 대기 단계로 복귀 가능.

#### 10.5.2 인게임 진행 중

- PlayerA/B가 나가면:
  - SC2 기본 규칙에 따라 남은 플레이어가 승리한다.
- Referee/Observer가 나가면:
  - 경기 규칙에는 영향 없음.
  - HUD 목록만 갱신한다.

---

### 10.6 심판 기능

심판 MVP 기능:

- 옵션 확정.
- 밴픽 시작.
- 밴픽 재설정.
- 진행 중단.
- 플레이어 Ready 상태 확인.

후순위 기능:

- 재경기 요청.
- 외부 결과 기록.
- 리플레이/결과 업로드 연동.

---

### 10.7 안전 장치

게임 시작 전 확인 조건:

- PlayerA/B 존재.
- PlayerA/B Ready 완료.
- 옵션 값 유효.
- `BanRounds[]` 생성 성공.
- `UnitMeta` 후보 풀 생성 성공.
- Ban 업그레이드/Requirement 검사 통과.

문제 발생 시:

- 개발 빌드:
  - Debug HUD와 로그에 상세 출력.
- 배포 빌드:
  - 사용자에게 간단한 오류 메시지 표시 후 옵션 단계로 복귀.

## 11. 에러/예외 시나리오

> 이 장에서는 외부 런처, BPM 맵 시작 전 단계, 밴픽, 밴 적용, 인게임 진행 중  
> 발생할 수 있는 주요 오류와 처리 방식을 정의한다.

---

### 11.1 외부 런처 단계 오류

#### 11.1.1 SC2 설치 경로 없음 / 잘못됨

- 상황
  - 런처가 SC2 실행 파일을 찾지 못함.
- 처리
  - “StarCraft II 설치 경로를 찾을 수 없습니다.” 메시지 표시.
  - 경로 선택 UI 제공.
  - 선택 경로를 `install-path.txt`에 저장.
- 재시도
  - 경로 설정 완료 후 파일 확인/실행 버튼 재활성화.

#### 11.1.2 필수 모드/맵 파일 누락

- 상황
  - `BPM_Core.SC2Mod` 또는 `BPM_1v1_*.SC2Map` 누락.
- 처리
  - 누락 파일 리스트 표시.
  - 설치/업데이트 버튼 활성화.
  - 필수 파일이 준비되기 전에는 실행 비활성화.

#### 11.1.3 다운로드/검증 실패

- 상황
  - ZIP 다운로드 실패, SHA256 불일치, ZIP 구조 오류.
- 처리
  - 설치 중단.
  - 캐시 파일 삭제 권장.
  - 재다운로드 버튼 제공.

---

### 11.2 맵 시작 전 단계 오류

#### 11.2.1 플레이어 부족

- 상황
  - Player 1 또는 Player 2가 없는 상태.
- 처리
  - “플레이어가 2명 필요합니다.” 메시지 표시.
  - Ready/밴픽 시작 버튼 비활성화.

#### 11.2.2 Host/심판 부재

- 상황
  - Host 또는 Referee가 나감.
- 처리
  - 심판이 있으면 심판 우선.
  - 없으면 Player 1 → Player 2 순서로 Host 재지정.
  - UI 권한 갱신.

#### 11.2.3 옵션 값 오류

- 상황
  - UnitPoolMode, BanCount, ShadowBan 옵션이 유효하지 않음.
- 처리
  - 기본값으로 복구.
  - 개발 빌드에서는 Debug HUD에 원인 출력.

---

### 11.3 밴픽 단계 오류

#### 11.3.1 유닛 풀/메타 데이터 누락

- 상황
  - `UnitMeta`에 없는 유닛이 후보 풀에 포함됨.
- 처리
  - 해당 유닛을 후보에서 제외.
  - 개발 빌드에서는 “UnitMeta 누락: <UnitId>” 로그 출력.

#### 11.3.2 잘못된 밴 입력

- 상황
  - 이미 밴된 유닛을 다시 선택하거나, 현재 차례가 아닌 플레이어가 입력함.
- 처리
  - 입력 무시.
  - 필요 시 “이미 밴된 유닛입니다.” 툴팁 표시.

#### 11.3.3 타이머 시간 초과

- 상황
  - 제한 시간 내에 밴 확정이 없음.
- 처리
  - 기본: 해당 라운드 패스.
  - 옵션: 무작위 밴 자동 선택.

---

### 11.4 밴 적용 단계 오류

#### 11.4.1 Ban 업그레이드 누락

- 상황
  - 특정 유닛에 대응하는 `BPM_Ban_UnitX` 업그레이드를 찾지 못함.
- 처리
  - 해당 유닛의 밴 적용을 스킵하지 말고 오류 상태로 기록.
  - 개발 빌드에서는 경고 출력.
  - 배포 빌드에서는 밴픽 시작 전 Consistency Check로 사전 차단하는 것이 목표.

#### 11.4.2 Requirement 연결 누락

- 상황
  - Ban 업그레이드는 지급됐지만 생산 버튼/변태/업그레이드가 잠기지 않음.
- 처리
  - 개발 빌드에서 `UnitMeta ↔ ButtonMeta ↔ Requirement` 검사 로그 출력.
  - 해당 유닛은 밴 후보에서 임시 제외 가능.

#### 11.4.3 우회 생산 경로

- 상황
  - 트리거 소환, 캠페인 패널, 용병, 변태 등으로 밴 유닛이 생성됨.
- 처리
  - 모든 생산 경로를 `ButtonMeta` 또는 별도 `SpawnPathMeta`로 관리.
  - 우회 경로 발견 시 해당 경로에도 동일 Ban Requirement 적용.

---

### 11.5 인게임 단계 오류

#### 11.5.1 시작 유닛 생성 실패

- 상황
  - 밴픽 완료 후 본진/일꾼이 생성되지 않음.
- 처리
  - 경기 시작을 중단하고 오류 메시지 표시.
  - 개발 빌드에서는 종족/시작 위치/생성 실패 유닛 출력.

#### 11.5.2 HUD 생성 실패

- 상황
  - 밴 HUD 또는 옵저버 HUD 생성 실패.
- 처리
  - HUD를 숨기고 게임은 계속 진행.
  - Debug HUD에 원인 출력.

#### 11.5.3 플레이어 이탈

- 밴픽 중:
  - 밴픽 중단 또는 이탈 플레이어 패배 처리.
- 인게임 중:
  - SC2 기본 규칙에 따라 남은 플레이어 승리.

---

### 11.6 개발/테스트용 진단 도구

- Debug HUD
  - 현재 BPMStage.
  - UnitPoolMode.
  - BanCount.
  - CurrentBanRoundIndex.
  - FinalDisabledUnits.
  - Host/Referee/Observer 매핑.
- Consistency Check
  - `UnitMeta`
  - `UpgradeMeta`
  - `ButtonMeta`
  - `Requirement`
  - 생산/변태/패널 경로
  간 연결을 검사한다.

## 12. 개발 단계 및 우선순위

> 수정된 로드맵은 **전용 런처 + BPM 전용 섬멸전 맵 내부 밴픽** 구조를 기준으로 한다.  
> 기존의 `런처 맵 → Bank → Next Map` 단계는 개발 대상에서 제외한다.

---

### 12.1 전체 로드맵 개요

1. **1단계 – 외부 런처 최소 골격**
2. **2단계 – 통합 모드 최소 골격 구축**
3. **3단계 – BPM 테스트 섬멸전 맵 + 시작 잠금 구조**
4. **4단계 – 인게임 옵션/Ready/심판/옵저버 기본 UI**
5. **5단계 – 밴픽 UI/라운드 로직**
6. **6단계 – 밴 적용(업그레이드/Requirement) + 인게임 HUD**
7. **7단계 – 캠페인+스타1 유닛 풀 추가**
8. **8단계 – 편의 기능(Convenience) 흡수/정리**
9. **9단계 – 유닛/맵 메타 데이터(DB) 정리 및 연동**
10. **10단계 – 맵 풀 확장 및 실제 래더 맵 포트**
11. **11단계 – 옵저버/GameHeart 스타일 확장 + 심판 UI 고도화**
12. **12단계 – 테스트/튜닝/배포 준비**

---

### 12.2 1단계 – 외부 런처 최소 골격

**목표**  
Mass Recall식 설치/실행 런처의 최소 기능을 만든다.

**작업 항목**

1. `BPM Launcher.exe` 프로젝트 생성.
2. SC2 설치 경로 자동 탐색.
3. `Support64\SC2Switcher_x64.exe` 탐색.
4. `Mods/`, `Maps/` 파일 존재 여부 확인.
5. 로컬 테스트 맵 직접 실행.
6. 상태 파일/설치 경로 저장.

**산출물**

- `BPM_1v1_TestMap.SC2Map`을 직접 실행할 수 있는 런처 MVP.

---

### 12.3 2단계 – 통합 모드 최소 골격 구축

**목표**  
`BPM_Core.SC2Mod`의 기본 구조와 공통 초기화/유틸을 만든다.

**작업 항목**

1. 새 모드 생성.
2. `Triggers/Init`, `Triggers/Util`, `Triggers/BanPick`, `Triggers/Balance`, `Triggers/Convenience` scaffold 작성.
3. `Data/DB`, `Data/Upgrades`, `Data/UI`, `Data/Units` scaffold 작성.
4. 기본 전역 변수 정의:
   - `CfgUnitPoolMode`
   - `CfgBanCount`
   - `BPMStage`
   - PlayerA/B/Observer/Referee 매핑.

---

### 12.4 3단계 – BPM 테스트 섬멸전 맵 + 시작 잠금 구조

**목표**  
`BPM_1v1_TestMap.SC2Map`에서 기본 3, 2, 1 이후에도 실제 플레이가 시작되지 않도록 만든다.

**작업 항목**

1. 테스트 맵 생성.
2. `BPM_Core.SC2Mod` 의존성 추가.
3. 시작 유닛 미생성 또는 제거.
4. 카메라/입력/자원 잠금.
5. 검은 배경 또는 전체 화면 Dialog 표시.
6. 버튼 클릭 시 시작 유닛을 생성하고 잠금 해제하는 테스트 트리거 작성.

**산출물**

- “가짜 시작 전 단계 → 실제 섬멸전 시작” 흐름 검증.

---

### 12.5 4단계 – 인게임 옵션/Ready/심판/옵저버 기본 UI

**목표**  
실제 섬멸전 맵 내부에 옵션/Ready/심판 UI를 만든다.

**작업 항목**

1. PlayerA/B/Observer/Referee 역할 판정.
2. Ready 버튼.
3. Host/심판 권한 처리.
4. UnitPoolMode/BanCount 옵션 UI.
5. 옵저버 읽기 전용 처리.
6. 모든 클라이언트 UI 동기화 확인.

---

### 12.6 5단계 – 밴픽 UI/라운드 로직

**목표**  
같은 맵 내부에서 밴픽을 수행한다.

**작업 항목**

1. 풀스크린 밴픽 UI.
2. 유닛 그리드.
3. `BanRounds[]` 생성.
4. 타이머/확정/패스 처리.
5. Public/Shadow/Self Ban 구조체 저장.
6. 옵저버/심판 관전 처리.

---

### 12.7 6단계 – 밴 적용 + 인게임 HUD

**목표**  
밴된 유닛이 실제로 생산/사용 불가능해지도록 한다.

**작업 항목**

1. `BPM_Ban_UnitX` 업그레이드 세트 정의.
2. Requirement 연결.
3. `FinalDisabledUnits` 계산.
4. 플레이어별 Ban 업그레이드 지급.
5. ME/ENEMY HUD 생성.
6. 생산/변태/업그레이드/패널 우회 경로 테스트.

---

### 12.8 7단계 – 캠페인+스타1 유닛 풀 추가

**목표**  
확장 유닛 풀을 실제 플레이 가능한 수준으로 추가한다.

**작업 항목**

1. 테란 프로토타입 유닛부터 추가.
2. 생산 건물/버튼/요구사항 연결.
3. 캠페인/스타1 유닛의 밸런스 기본값 설정.
4. UnitPoolMode에 따라 후보 그리드와 생산 가능 여부 분기.
5. 이후 저그/프로토스 확장.

---

### 12.9 8단계 이후

- 8단계: 편의 기능 흡수.
- 9단계: DB 정리 및 연동.
- 10단계: 실제 래더 맵 포트.
- 11단계: 옵저버/GameHeart + 심판 UI 고도화.
- 12단계: 런처 설치/업데이트, Battle.net 사용자 지정 게임, 밴픽/밴 적용/맵별 회귀 테스트.
